# 구현 리포트 — Serving 에러 처리 견고화 + Workers 노출

## 배경

이전 deep-explain 분석에서 MMP 서빙 API에 대해 두 가지 개선을 도출했다.
1. `_handle_prediction_error`의 키워드 기반 매칭을 타입 기반으로 전환
2. 에러 응답 본문 구조화 (additive — 기존 `detail` 필드 보존)
3. `run_api_server`에서 다중 워커 옵션 노출

본 작업은 이 세 항목을 적용한다. spec 문서는 별도로 없으며, deep-explain 분석 보고서가 사실상의 spec.

## 변경 목록

- `mmp/serving/router.py`
  - `import` 추가: `Optional` (typing), `MlflowException` (mlflow.exceptions)
  - `_MLFLOW_CLIENT_ERROR_CODES = frozenset({"INVALID_PARAMETER_VALUE", "BAD_REQUEST", "INVALID_STATE"})` — 클라이언트 입력 오류로 분류할 MLflow error_code 화이트리스트
  - `_handle_prediction_error` 본문 재작성: 키워드 매칭 제거, `isinstance(e, MlflowException)` + `error_code` 화이트리스트로 422/500 분기. 다른 모든 예외는 타입 이름과 함께 로깅 후 500
  - `_build_error_body(status_code, detail, error_type, request_id)` 신규: 응답 본문을 `{detail, error_code, message, error_type?, request_id?}` 형태로 통일. `detail`은 FastAPI 컨벤션 호환을 위해 보존
  - `@app.exception_handler(HTTPException)` `http_exception_handler` 신규: 모든 HTTPException을 `_build_error_body` 결과로 변환. `X-Request-ID` 헤더가 있으면 본문에 포함, 응답 헤더(WWW-Authenticate 등)는 보존
  - `@app.exception_handler(Exception)` `unhandled_exception_handler` 신규: 라우트가 잡지 못한 예외에 대한 마지막 안전망. error_type을 본문에 포함하고 항상 500 반환
  - `run_api_server`: workers 추출 + 분기 로직 추가. `getattr` 체인으로 `settings.config.serving.workers`를 안전하게 읽고, `isinstance(int) and ≥ 1` 검사 통과 시에만 사용. workers > 1이면 `uvicorn.run("mmp.serving.router:app", workers=N)` (import string), 그 외는 기존 동작 그대로 `uvicorn.run(app, host, port)` 유지
- `tests/unit/serving/test_router.py`
  - `TestPredictionErrorHandling` 클래스 신규 (4 테스트):
    - `test_mlflow_invalid_parameter_value_returns_422` — INVALID_PARAMETER_VALUE는 422
    - `test_mlflow_other_error_code_returns_500` — INTERNAL_ERROR 등은 500
    - `test_unexpected_exception_returns_500` — RuntimeError 등은 500
    - `test_predict_endpoint_mlflow_input_error_returns_422` — 라우트 통합 검증 (mock predict가 MlflowException 발생)
  - `TestStructuredErrorResponse` 클래스 신규 (3 테스트):
    - `test_error_response_includes_structured_fields` — 503 응답에 detail/error_code/message 모두 존재
    - `test_error_response_includes_request_id` — X-Request-ID가 본문과 응답 헤더에 모두 포함
    - `test_unhandled_exception_returns_500_with_error_type` — handlers.health가 raise하면 글로벌 핸들러가 error_type 포함 500 반환
  - `TestRunAPIServerWorkers` 클래스 신규 (3 테스트):
    - `test_single_worker_uses_app_object` — workers=1은 app 객체 전달
    - `test_multi_worker_uses_import_string` — workers=4는 import string + workers=4
    - `test_invalid_workers_falls_back_to_single` — 정수가 아닌 값은 1로 폴백

## spec 대비 검토

| 분석 보고서 결정 | 구현 상태 | 비고 |
|---|---|---|
| 키워드 매칭 → MlflowException 타입 매칭 | 구현 완료 | error_code 화이트리스트 방식. 메시지 텍스트 변경에 면역 |
| 에러 응답 구조화 (additive) | 구현 완료 | `detail` 보존 + `error_code`/`message`/`error_type`/`request_id` 추가 |
| Solace식 20+ 예외 계층 도입 | 의도적 미구현 | 분석 보고서에서 "도메인 성숙도 불일치"로 명시적 제외. 작은 표적 수정만 적용 |
| run_api_server에 workers 노출 | 구현 완료 | `workers > 1`일 때만 import string 경로. `workers=1` 기본 경로는 기존 동작 비파괴 보존 |
| AnyIO 스레드풀 토큰 조정 | 의도적 미구현 | 우선순위 4로 분류, 본 PR 범위 외 |

## 회귀 수정 (사용자 추가 요청)

본 PR 1차 작업 후, 사용자의 사전 미커밋 `_endpoints.py` 변경에서 비롯된 통합 테스트 8건의
회귀를 별도 작업 단위로 진단하고 수정했다.

### 진단

`_endpoints.predict()`의 추출 로직(`predictions_df.shape[1] > 1`로 단일/다중 출력 분기)이
`format_predictions`가 부착하는 메타데이터 컬럼(entity_columns/timestamp_column/treatment_column)을
*예측 컬럼으로 오인*했다. 단일 출력 분류 모델에 entity_id가 부착되면 shape==2가 되어
다중 출력 dict로 잘못 분류되어 `MinimalPredictionResponse.prediction`이 스칼라가 아닌
`{prediction: 1.0, entity_id: 42}` 형태로 반환됐다. 통합 테스트 8건이 모두
`assert isinstance(pred_data["prediction"], (int, float))`에서 실패.

배치 엔드포인트는 `predictions_df.to_dict(orient="records")`로 모든 컬럼을 의도적으로
포함시키므로(entity join 목적) 영향 없음. 수정은 단일 엔드포인트(`predict()`)에만 적용.

### 수정

- `mmp/serving/_endpoints.py`에 두 헬퍼 추가:
  - `_metadata_columns(schema)`: data_interface 메타데이터 컬럼 이름 집합 추출
  - `_strip_metadata_columns(df, schema)`: 메타데이터 제거된 view 반환
- `predict()` 추출 로직이 `prediction_only_df`(메타데이터 제거 view)를 기준으로 분기
- `predict_batch()`는 전혀 수정하지 않음 (의도적 모든 컬럼 포함 동작 보존)

### 회귀 방지 테스트 5건 추가

- `test_single_output_with_entity_id_returns_scalar` — entity_id 부착 단일 출력 → 스칼라

## 번외 발견

- `run_api_server`가 `--reload` 옵션을 노출하지 않음. 개발 환경에서 hot-reload가 필요하면 향후 추가 가치 있음. 본 PR 범위 외.
- `test_run_api_server_enabled_serving`이 `mock_settings.serving = ...` (config 경로 없음)으로 설정하는데, 실제 코드는 `settings.config.serving.enabled`를 확인한다. Mock 자동 속성 덕에 우연히 통과 중. 이 테스트의 자체 설계는 정확하지 않지만 본 PR 범위 외라 그대로 둠.
- `format_predictions`(`mmp/utils/data/data_io.py:501`)가 모든 결과에 메타데이터를 부착하는 동작은 inference 파이프라인의 join 요구 때문이고 의도적 설계로 판단됨. API 서빙의 단일 엔드포인트가 이를 다른 의미로 해석하는 것이 문제였으므로 수정은 호출자(`_endpoints.predict`) 측에 적용하는 것이 옳음.
